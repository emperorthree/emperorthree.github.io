# 索引问题
## 索引的常见数据存储模块
1. 哈希表

   **区间查询慢只能全表扫描**

2. 有序数组

   插入数据效率低

## 3. 搜索树

![](http://qc0iyzbq5.bkt.clouddn.com/04fb9d24065635a6a637c25ba9ddde68.png)

二叉搜索树的特点：左节点 < 根节点 < 右节点 查询和插入时间复杂度都是logn

原则上，树可以存在多叉只要保持左到右和上到下的大小序列。但二叉树的搜索效率是最高的，**可大多数数据库存储还是不会采用二叉树的结构，因为索引不止在内存中也会放在硬盘上。**

> 你可以想象一下一棵100万节点的平衡二叉树，树高20。一次查询可能需要访问20个数据块。在机械硬盘时代，从磁盘随机读一个数据块需要10 ms左右的寻址时间。也就是说，对于一个100万行的表，如果使用二叉树来存储，单独访问一个行可能需要20个10 ms的时间，这个查询可真够慢的。
>
> 为了让一个查询尽量少地读磁盘，就必须让查询过程访问尽量少的数据块。那么，我们就不应该使用二叉树，而是要使用“N叉”树。这里，“N叉”树中的“N”取决于数据块的大小。

> 你心里要有个概念，数据库底层存储的核心就是基于这些数据模型的。每碰到一个新数据库，我们需要先关注它的数据模型，这样才能从理论上分析出这个数据库的适用场景。

## InnoDB的索引模型

* b+树
* 主键索引也被称为聚簇索引、非主键索引也被称为二级索引
* 页分裂、页合并

![](http://qc0iyzbq5.bkt.clouddn.com/dcda101051f28502bd5c4402b292e38d.png)

* 主键索引的叶子节点直接是对应主键的row内容，而非主键索引节点是主键值；由此可以看出主键索引一旦建立最好是不要去更改，会影响到多有的非主键索引。

* 页分裂和合并出现在维护索引表的时候，当需要插入的值是400时，需要向后移空出空间来，但刚好这个数据页也正好没有空间，就会发生页分裂。根据B+树的算法，这时候需要申请一个新的数据页，然后挪动部分数据过去。这个过程称为页分裂。在这种情况下，性能自然会受影响。整体空间利用率降低约50%。
* 当页使用率过于低就会发生页合并
* 选择自增主键的优点，主键长度小，普通索引的叶子节点就越小，占用空间也就相对小了。

* **最左前缀原则**

  根据B+树的结构具备这种特性，对于复合索引B+树会按照索引字段的顺序排列，满足联合索引的最左N个字段或者最左N个字符串都可以命中索引。

* **建立联合索引原则**
  * 第一原则是尽量通过索引字段顺序的优化减少索引表的维护
  * 第二是通过第一条无法满足查询条件，就要考虑空间使用字段长度较小的建立单独索引

* **索引下推**

  前提是能命中索引，对于使用到索引的条件，进行满足条件的筛选，剩下的是没有建立索引的条件通过回表得到字段值在进行判断。

```sql
CREATE TABLE `tuser` (
  `id` int(11) NOT NULL,
  `id_card` varchar(32) DEFAULT NULL,
  `name` varchar(32) DEFAULT NULL,
  `age` int(11) DEFAULT NULL,
  `ismale` tinyint(1) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `id_card` (`id_card`),
  KEY `name_age` (`name`,`age`)
) ENGINE=InnoDB

select * from tuser where name like '张%' and age=10 and ismale=1;
```

* **索引重建**

  由于会出现数据页分裂等问题造成空间利用率降低，所以需要索引重建。（索引可能因为删除，或者页分裂等原因，导致数据页有空洞）

  ```sql
  alter table T drop index k; 
  alter table T add index(k);
  ```

  k为非主键索引这样重建是合理的，若k为主键索引则不建议这样重建。

  **重建主键索引可以用这个命令代替**

  ```sql
  alter table T engine=InnoDB
  ```

## 字符串字段加索引的方式

* 使用好前缀索引
* 避免长字符串索引的创建太占空间
* 身份证号倒序存储+前缀索引构建
* 前缀索引也会来带回表的消耗
* 倒序存储，再创建前缀索引，用于绕过字符串本身前缀的区分度不够的问题；
* 创建hash字段索引，查询性能稳定，有额外的存储和计算消耗，跟第三种方式一样，都不支持范围扫描。

## 普通索引和唯一索引

* 判断

  对于普通索引来说，查找到满足条件的第一个记录(5,500)后，需要查找下一个记录，**直到碰到第一个不满足k=5条件的记录**。

  对于唯一索引来说，由于索引定义了唯一性，**查找到第一个满足条件的记录后，就会停止继续检索**。

  但是这个性能的代价是十分微小的，因为innodb是按页来读取数据到内存的，但如果这条被判断的记录不在当前页就会麻烦一点，这种概率也十分微小。

* **change buffer**

  当需要更新一个数据页时，如果数据页在内存中就直接更新，而如果这个数据页还没有在内存中的话，在不影响数据一致性的前提下，InooDB会将这些更新操作缓存在change buffer中，这样就不需要从磁盘中读入这个数据页了。在下次查询需要访问这个数据页的时候，将数据页读入内存，然后执行change buffer中与这个页有关的操作。通过这种方式就能保证这个数据逻辑的正确性。

* 什么条件下会使用change buffer？

  更新不需要从磁盘读取数据判断更新条件；

  唯一索引插入记录需要先将数据页读到内存判断插入值是否会违反唯一约束；判断完直接在内存更新；故**唯一索引就不会使用change buffer；故只有普通索引可以使用。**

  * 记录更新的目标在内存中

    这种情况两者差别不大，只不过唯一索引会多一个判断的过程，性能消耗微乎其微；

  * 记录更新的目标不在内存中

    普通索引减少了从磁盘中读取数据页的过程，减少了io操作的消耗，通过change buffer加快操作执行

* merge

  change buffer通过将变更记录缓存下来，当查询被变更数据时会触发merge，数据库定时也会去触发merge；目的是增加一次merge的收益，减少频繁的io；

  不难看出当更新的数据马上就会被查找时使用change buffer不仅不能达到效果可能还会适得其反

* change buffer用的是buffer pool里的内存，因此不能无限增大。change buffer的大小，可以通过参数**innodb_change_buffer_max_size**来动态设置。这个参数设置为50的时候，表示change buffer的大小最多只能占用buffer pool的50%。

**redo log 主要节省的是随机写磁盘的IO消耗（转成顺序写），而change buffer主要节省的则是随机读磁盘的IO消耗。**

## 为什么会选错索引？

* 优化器
* 评判标准
* 索引基数
* 解决办法

### 优化器选择索引过程

索引的选择是优化器的工作，通过预估需要扫描多少行加上读取数据页的消耗综合判断出使用合理的索引，这里就可以做一个说明，并不是预估扫描行数少就一定会去走那个索引的不要忘了非主键索引还需要加上回表IO的消耗。

* 预估扫描rows是如何判断的？

答：基数，索引基数越大索引的区分度越高；SHOW INDEX FROM TABLENAME 显示表的索引信息其中cardinality参数就是基数值；基数是通过“采样统计”计算出来，随机抽取N个页统计这些页不同值得到一个平均值，然后乘以这个索引的页面数；数据在持续更新的时候索引统计信息也会改变，当数据变更行数超过1/M时，会触发重新统计索引。

在MySQL中，有两种存储索引统计的方式，可以通过设置参数innodb_stats_persistent的值来选择：

- 设置为on的时候，表示统计信息会持久化存储。这时，默认的N是20，M是10。
- 设置为off的时候，表示统计信息只存储在内存中。这时，默认的N是8，M是16

### 解决选错索引的问题

1. 索引表统计信息有误，（结果预估的rows值跟实际情况差距比较大）

   analyze table tablename；//重新统计索引信息

2. select * from t where (a between 1 and 1000)  and (b between 50000 and 100000) order by b limit 1;分析这个语句，a、b上都建立了索引；该语句执行选着了b索引，显然如果选择a索引在通过b的条件来过滤会更快避免了回表的操作；

   问题：为什么会选错？怎么去解决？

答：先看解决：①：使用force index强制走a索引；不优雅，索引名称改动、迁库都会有影响；②：修改语句引导MySQL选择正确的索引；上述sql语句之所以选择b索引是为了基于索引的有序性避免排序；order by b limit 1=》order by b,a limit 1因为有limit1逻辑上一致，才可这样修改；③：新建更合适的索引，删掉误用的索引；
